## Overview

The root-level Next.js app is the current application foundation. It is a Next.js App Router project generated by v0 with a working login page, dashboard shell, sidebar navigation, operational dashboard widgets, inventory pages, CSV import UI, reorder UI, supplier orders UI, forecasting UI, audit log, settings, shadcn-style UI primitives, mock data, and brand assets.

The architecture goal is to evolve this scaffold incrementally into a maintainable frontend-first internal restaurant inventory platform without rewriting the shell or introducing backend complexity prematurely.

## Messy or Redundant Parts

| Issue | Impact | Recommended Fix |
|-------|--------|-----------------|
| Inventory category mismatch | `inventory/page.tsx` filters `Ingredients` and `Tools`, but mock categories are `Meats`, `Seafood`, `Vegetables`, `Supplies`, and `Sauces`; tabs will not represent data correctly | Normalize category model before expanding inventory |
| Page-local mock arrays | CSV, forecasting, item detail, alerts, and audit pages define data locally | Move into typed feature fixtures when those pages are touched |
| Repeated table shells | Inventory, supplies, orders, audit, and recommendations repeat borders, headers, empty states, and action cells | Extract shared `DataTableShell` or small table composition helpers |
| Fixed desktop sidebar | `pl-64` layout is not iOS/tablet friendly | Add responsive sidebar/mobile nav without replacing the shell |
| Overclaiming integrations | Future integrations can be mistaken for active behavior if copy is not explicit | Label placeholders accurately until real integrations exist |
| Generic restaurant content | Generic ingredients and vendor examples weaken the Sonsie-specific demo | Keep mock data aligned to Sonsie operations |
| Large page files | Orders, settings, forecasting, and CSV import are long client components | Extract feature components gradually |
| Duplicate globals | `styles/globals.css` duplicates default shadcn theme and is likely unused | Remove or clearly mark unused after verification |

## Mock Data Strategy

Current mock data lives mostly in `lib/mock-data.ts`, with some page-local arrays. That is acceptable for v0 output but should be normalized before building more features.

Unless the project maintainers explicitly request otherwise, every iteration should keep website data input/output mocked. New or refactored features should load their sample data from centralized fixture lists, typed mock modules, JSON-like mock structures, or `.txt` sample data files. Do not hardcode data directly in route pages or UI components. Keep the mock data boundary open-ended because the final CSV, backend, supplier, forecasting, and auth data flows are not set yet.

Recommended structure:

```
lib/mock/
|-- activity.ts
|-- forecasting.ts
|-- inventory.ts
|-- orders.ts
|-- recommendations.ts
|-- restaurant-context.ts
|-- samples/
|   |-- daily-order-history.txt
|   `-- supplier-catalog.txt
|-- suppliers.ts
`-- index.ts
```

Rules:

- Keep mock data typed with exported domain types.
- Use Sonsie-relevant item names and categories.
- Model inventory categories as `ingredient`, `supply`, and `tool`, with optional subcategories like `meat`, `banchan`, `sauce`, `disposable`, `serviceware`, `grill_tool`, and `equipment`.
- Keep status calculation helpers separate from fixture arrays.
- Keep mock user actions obvious; do not fake persistence silently.

## Future API and Integration Boundaries

Do not implement these until requested, but keep frontend boundaries ready.

| Future Boundary | Suggested Module | Responsibility |
|-----------------|------------------|----------------|
| Auth | `features/auth`, later `server/auth` | Password gate now, real sessions/RBAC later |
| CSV ingestion | `features/ingest`, later `app/api/ingest/csv/route.ts` | Upload, preview, validate, commit order rows |
| Sysco | `features/suppliers`, later `server/integrations/sysco` | Catalog matching, draft order submission, sync status |
| Forecasting | `features/forecasting`, later `server/services/forecasting` | Forecast runs, usage projections, confidence explanations |
| Inventory persistence | `features/inventory`, later `server/repositories/inventory` | Inventory reads, adjustments, item detail history |
| Audit | `features/activity`, later `server/repositories/audit` | Immutable activity events for imports, edits, orders |

## Refactor Strategy

1. Normalize inventory categories and mock data first.
2. Extract reusable inventory/supplies table pieces that directly improve current pages.
3. Add responsive shell behavior without replacing the sidebar/header system.
4. Move page-local mock arrays into feature mock modules only when editing those pages.
5. Rework dashboard content toward operational readiness: low stock, supplies visibility, reorder queue, CSV import status, and recent activity.
6. Add new screens as frontend mock workflows before any backend wiring.
